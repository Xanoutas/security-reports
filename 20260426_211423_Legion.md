# Security Audit Report: Legion Protocol

## Executive Summary

This security audit was conducted on the Legion Protocol, a token sale and vesting platform. The audit identified several critical vulnerabilities including precision loss in token allocation calculations, dangerous strict equality comparisons that can lead to DoS attacks, and reentrancy vulnerabilities in token claiming functions.

The most critical issues involve:
- Precision loss during token allocation calculations that can lead to incorrect token distributions
- Strict equality checks that can be exploited to prevent legitimate operations
- Reentrancy vulnerabilities in vesting contract creation

## High Severity Findings

### H-1: Precision Loss in Token Allocation Calculation

**Severity:** High  
**Location:** `LegionPreLiquidApprovedSale.claimTokenAllocation()` - Line 397-480  

**Description:**
The contract performs division before multiplication in the token allocation calculation, leading to significant precision loss. The calculation `totalAmount = _saleStatus.askTokenTotalSupply * position.cache` uses `position.cache` which is derived from a previous division operation, causing truncation errors.

**Impact:**
- Users may receive fewer tokens than entitled due to precision loss
- Token distributions become unfair and inconsistent
- Financial loss for investors in token sales

**Proof of Concept:**
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.30;

import "forge-std/Test.sol";

contract PrecisionLossTest is Test {
    function testPrecisionLoss() public {
        // Simulate the precision loss scenario
        uint256 askTokenTotalSupply = 1000000e18; // 1M tokens
        uint256 userContribution = 333; // User contributed 333 wei
        uint256 totalContributions = 1000; // Total contributions 1000 wei
        
        // Current implementation (divide then multiply)
        uint256 cache = userContribution / totalContributions; // This equals 0 due to truncation
        uint256 tokensReceived = askTokenTotalSupply * cache; // 1000000e18 * 0 = 0
        
        // Correct implementation (multiply then divide)
        uint256 correctTokens = (askTokenTotalSupply * userContribution) / totalContributions;
        
        console.log("Tokens with current implementation:", tokensReceived);
        console.log("Tokens with correct implementation:", correctTokens);
        
        // User receives 0 tokens instead of 333000e18 tokens
        assertEq(tokensReceived, 0);
        assertEq(correctTokens, 333000e18);
        assert(tokensReceived != correctTokens);
    }
}
```

**Recommendation:**
Modify the calculation to perform multiplication before division:
```solidity
totalAmount = (_saleStatus.askTokenTotalSupply * userContribution) / totalContributions;
```

### H-2: DoS Attack via Strict Equality Check in Token Supply Verification

**Severity:** High  
**Location:** `LegionPreLiquidApprovedSale._verifyCanSupplyTokens()` - Line 917  

**Description:**
The function uses a dangerous strict equality check `totalTokensAllocated == 0` which can be exploited by an attacker to prevent legitimate token supply operations. An attacker can make minimal allocations to ensure the total is never exactly zero.

**Impact:**
- Complete denial of service for token supply operations
- Protocol functionality can be permanently broken
- Legitimate users cannot interact with the contract

**Proof of Concept:**
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.30;

import "forge-std/Test.sol";

contract MockSale {
    uint256 public totalTokensAllocated;
    
    function allocateTokens(uint256 amount) external {
        totalTokensAllocated += amount;
    }
    
    function _verifyCanSupplyTokens() internal view {
        require(totalTokensAllocated == 0, "Tokens already allocated");
    }
    
    function supplyTokens() external {
        _verifyCanSupplyTokens();
        // Token supply logic here
    }
}

contract DoSAttackTest is Test {
    MockSale sale;
    
    function setUp() public {
        sale = new MockSale();
    }
    
    function testDoSAttack() public {
        // Attacker makes minimal allocation to break strict equality
        sale.allocateTokens(1);
        
        // Now legitimate token supply will always fail
        vm.expectRevert("Tokens already allocated");
        sale.supplyTokens();
        
        // Even if most tokens are withdrawn, as long as totalTokensAllocated != 0,
        // the function will fail
        assertTrue(sale.totalTokensAllocated() != 0);
    }
}
```

**Recommendation:**
Replace strict equality with a more appropriate check:
```solidity
require(totalTokensAllocated <= maxAllowableRemainder, "Too many tokens allocated");
```

### H-3: DoS Attack via Strict Equality in Capital Withdrawal

**Severity:** High  
**Location:** `LegionCapitalRaise._verifyCanWithdrawCapital()` - Line 655  

**Description:**
Similar to H-2, this function uses strict equality `capitalRaiseStatus.totalCapitalRaised == 0` which can be exploited to prevent capital withdrawal operations.

**Impact:**
- Prevention of legitimate capital withdrawals
- Funds can become stuck in the contract
- Complete disruption of capital raise functionality

**Proof of Concept:**
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.30;

import "forge-std/Test.sol";

contract MockCapitalRaise {
    struct CapitalRaiseStatus {
        uint256 totalCapitalRaised;
    }
    
    CapitalRaiseStatus public capitalRaiseStatus;
    
    function contribute(uint256 amount) external {
        capitalRaiseStatus.totalCapitalRaised += amount;
    }
    
    function _verifyCanWithdrawCapital() internal view {
        require(capitalRaiseStatus.totalCapitalRaised == 0, "Capital already raised");
    }
    
    function withdrawCapital() external {
        _verifyCanWithdrawCapital();
        // Withdrawal logic here
    }
}

contract CapitalDoSTest is Test {
    MockCapitalRaise capitalRaise;
    
    function setUp() public {
        capitalRaise = new MockCapitalRaise();
    }
    
    function testCapitalWithdrawalDoS() public {
        // Attacker contributes minimal amount
        capitalRaise.contribute(1);
        
        // Now capital withdrawal is permanently blocked
        vm.expectRevert("Capital already raised");
        capitalRaise.withdrawCapital();
    }
}
```

**Recommendation:**
Replace with a threshold-based check or use proper state management.

## Medium Severity Findings

### M-1: Reentrancy Vulnerability in Token Allocation Claims

**Severity:** Medium  
**Location:** Multiple locations including `LegionTokenDistributor.claimTokenAllocation()` - Line 136-195  

**Description:**
The token claiming functions are vulnerable to reentrancy attacks during vesting contract creation. External calls to `_createVesting()` occur before state updates, allowing potential reentrancy.

**Impact:**
- Potential for multiple claims of the same allocation
- Unexpected behavior during vesting contract creation
- Possible exploitation if vesting factory is compromised

**Proof of Concept:**
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.30;

import "forge-std/Test.sol";

contract ReentrancyTest is Test {
    MockTokenDistributor distributor;
    MaliciousVestingFactory maliciousFactory;
    
    function setUp() public {
        maliciousFactory = new MaliciousVestingFactory();
        distributor = new MockTokenDistributor(address(maliciousFactory));
    }
    
    function testReentrancyAttack() public {
        // Set up malicious factory to call back into distributor
        maliciousFactory.setTarget(address(distributor));
        
        // This could potentially be exploited for reentrancy
        distributor.claimTokenAllocation(1);
        
        // Verify the reentrancy attempt was made
        assertTrue(maliciousFactory.reentrancyAttempted());
    }
}

contract MockTokenDistributor {
    address vestingFactory;
    mapping(uint256 => bool) public claimed;
    
    constructor(address _vestingFactory) {
        vestingFactory = _vestingFactory;
    }
    
    function claimTokenAllocation(uint256 allocationId) external {
        // Vulnerable: external call before state update
        address vestingAddress = _createVesting();
        
        // State update happens after external call
        claimed[allocationId] = true;
    }
    
    function _createVesting() internal returns (address) {
        return MaliciousVestingFactory(vestingFactory).createVesting();
    }
}

contract MaliciousVestingFactory {
    address target;
    bool public reentrancyAttempted;
    
    function setTarget(address _target) external {
        target = _target;
    }
    
    function createVesting() external returns (address) {
        if (target != address(0)) {
            reentrancyAttempted = true;
            // Attempt reentrancy (would fail due to state, but demonstrates vulnerability)
            try MockTokenDistributor(target).claimTokenAllocation(2) {} catch {}
        }
        return address(0x123);
    }
}
```

**Recommendation:**
Implement the checks-effects-interactions pattern and add reentrancy guards:
```solidity
modifier nonReentrant() {
    require(!_locked, "Reentrant call");
    _locked = true;
    _;
    _locked = false;
}

function claimTokenAllocation(...) external nonReentrant {
    // Update state first
    claimed[allocationId] = true;
    
    // Then make external calls
    vestingAddress = _createVesting(...);
}
```

## Low/Informational Findings

### L-1: Missing Input Validation

**Description:** Several functions lack proper input validation for addresses and amounts.

**Recommendation:** Add comprehensive input validation including zero address checks and reasonable bounds checking.

### L-2: Lack of Event Emissions

**Description:** Critical state changes are not properly logged through events.

**Recommendation:** Add appropriate events for all significant state changes to improve transparency and monitoring.

### L-3: Gas Optimization Opportunities

**Description:** Some loops and storage operations could be optimized for gas efficiency.

**Recommendation:** Consider batching operations and optimizing storage layouts.

## Conclusion

The Legion Protocol contains several critical vulnerabilities that require immediate attention. The precision loss issue (H-1) and strict equality DoS vulnerabilities (H-2, H-3) pose significant risks to protocol functionality and user funds. The reentrancy issues (M-1) could potentially be exploited under certain conditions.

**Priority Recommendations:**
1. **Immediate:** Fix precision loss calculations and remove dangerous strict equality checks
2. **High Priority:** Implement reentrancy protections across all claiming functions
3. **Medium Priority:** Address input validation and event emission issues

All high and medium severity issues should be resolved before any production deployment or mainnet launch.
