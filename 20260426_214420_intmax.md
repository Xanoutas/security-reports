# INTMAX Protocol Security Assessment Report

## Executive Summary

This security assessment was conducted on the INTMAX protocol, focusing on the token minting system, block builder rewards, and associated smart contracts. The audit identified **2 High severity** vulnerabilities and **2 Medium severity** issues that require immediate attention.

The most critical findings include:
- Unchecked token transfers that can lead to silent failures
- Reentrancy vulnerabilities in the minting mechanism
- Precision loss in reward calculations
- Dangerous strict equality checks

## High Severity Findings

### H-01: Unchecked Token Transfer Can Lead to Silent Failures

**Severity:** High  
**Location:** `src/block-builder-reward/BlockBuilderReward.sol#145`

**Description:**
The `claimReward` function in `BlockBuilderReward.sol` performs a token transfer without checking the return value. This can lead to silent failures where users believe they have successfully claimed rewards, but the transfer actually failed.

```solidity
function claimReward(uint256 reward) external {
    // ... validation logic ...
    intmaxToken.transfer(_msgSender(), reward); // @audit unchecked transfer
}
```

**Impact:**
- Users may lose rewards due to silent transfer failures
- Contract state becomes inconsistent with actual token balances
- Loss of user funds and protocol integrity

**Proof of Concept:**
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "../src/block-builder-reward/BlockBuilderReward.sol";
import "../src/token/scroll/ScrollINTMAXToken.sol";

contract UncheckedTransferExploit is Test {
    BlockBuilderReward public reward;
    ScrollINTMAXToken public token;
    address public admin = makeAddr("admin");
    address public rewardManager = makeAddr("rewardManager");
    address public contributionContract = makeAddr("contribution");
    address public attacker = makeAddr("attacker");

    function setUp() public {
        vm.startPrank(admin);
        
        // Deploy token
        token = new ScrollINTMAXToken();
        token.initialize(admin, admin, 1000000e18);
        
        // Deploy reward contract
        reward = new BlockBuilderReward();
        reward.initialize(admin, rewardManager, contributionContract, address(token));
        
        vm.stopPrank();
    }

    function testUncheckedTransferFailure() public {
        vm.startPrank(rewardManager);
        
        // Set a reward for the attacker
        reward.setReward(attacker, 1000e18);
        
        vm.stopPrank();
        vm.startPrank(attacker);
        
        // Record initial balance
        uint256 initialBalance = token.balanceOf(attacker);
        
        // Attempt to claim reward (will fail silently if contract has insufficient balance)
        reward.claimReward(1000e18);
        
        // Balance remains unchanged despite "successful" claim
        assertEq(token.balanceOf(attacker), initialBalance);
        
        // But reward is marked as claimed
        assertEq(reward.pendingReward(attacker), 0);
    }
}
```

**Recommendation:**
Use OpenZeppelin's `SafeERC20` library or check the return value:

```solidity
import "@openzeppelin/contracts/token/ERC20/utils/SafeERC20.sol";

function claimReward(uint256 reward) external {
    // ... validation logic ...
    SafeERC20.safeTransfer(IERC20(intmaxToken), _msgSender(), reward);
}
```

### H-02: Reentrancy Vulnerability in Minter Contract

**Severity:** High  
**Location:** `src/minter/Minter.sol#58-79`

**Description:**
The `mint` function in the Minter contract is vulnerable to reentrancy attacks. The function reads the balance before an external call to `intmaxToken.mint()`, which could be exploited if the token contract allows callbacks during minting.

```solidity
function mint() external {
    uint256 balanceBefore = intmaxToken.balanceOf(address(this)); // @audit balance read before external call
    
    intmaxToken.mint(address(this)); // @audit external call allowing reentrancy
    
    uint256 balanceAfter = intmaxToken.balanceOf(address(this));
    uint256 mintedAmount = balanceAfter - balanceBefore;
    
    if (mintedAmount == 0) { // @audit dangerous strict equality
        revert NoTokensToMint();
    }
    
    // ... rest of function
}
```

**Impact:**
- Attacker could manipulate balance calculations through reentrancy
- Potential for double-spending or incorrect minting amounts
- Contract state corruption

**Proof of Concept:**
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "../src/minter/Minter.sol";
import "../src/token/mainnet/INTMAXToken.sol";

contract MaliciousToken {
    mapping(address => uint256) public balanceOf;
    address public minter;
    uint256 public reentrantCalls;
    
    constructor() {
        balanceOf[address(this)] = 1000000e18;
    }
    
    function setMinter(address _minter) external {
        minter = _minter;
    }
    
    function mint(address to) external {
        if (reentrantCalls < 2 && msg.sender == minter) {
            reentrantCalls++;
            // Reenter the mint function
            Minter(minter).mint();
        }
        balanceOf[to] += 100e18;
    }
    
    function transfer(address to, uint256 amount) external returns (bool) {
        balanceOf[msg.sender] -= amount;
        balanceOf[to] += amount;
        return true;
    }
}

contract ReentrancyExploit is Test {
    Minter public minter;
    MaliciousToken public maliciousToken;
    address public admin = makeAddr("admin");
    address public liquidity = makeAddr("liquidity");

    function setUp() public {
        maliciousToken = new MaliciousToken();
        
        minter = new Minter();
        minter.initialize(address(maliciousToken), liquidity, admin);
        
        maliciousToken.setMinter(address(minter));
    }

    function testReentrancyExploit() public {
        uint256 initialBalance = maliciousToken.balanceOf(address(minter));
        
        // Trigger reentrancy
        minter.mint();
        
        uint256 finalBalance = maliciousToken.balanceOf(address(minter));
        
        // Due to reentrancy, more tokens may have been processed than expected
        console.log("Reentrancy calls:", maliciousToken.reentrantCalls());
        console.log("Balance change:", finalBalance - initialBalance);
    }
}
```

**Recommendation:**
Implement reentrancy protection using OpenZeppelin's `ReentrancyGuard`:

```solidity
import "@openzeppelin/contracts/security/ReentrancyGuard.sol";

contract Minter is ReentrancyGuard {
    function mint() external nonReentrant {
        // ... existing logic
    }
}
```

## Medium Severity Findings

### M-01: Division Before Multiplication Causes Precision Loss

**Severity:** Medium  
**Location:** `src/token/mainnet/INTMAXToken.sol#119-137`

**Description:**
The `totalMintableAmount` function performs division before multiplication, leading to precision loss in reward calculations. This affects the accuracy of token minting amounts over time.

```solidity
function totalMintableAmount() public view returns (uint256) {
    uint256 elapsedDays = (block.timestamp - GENESIS_TIMESTAMP) / 86400; // @audit division first
    // ... 
    totalReward += elapsedDays * rewardPerDay; // @audit then multiplication
}
```

**Impact:**
- Systematic underestimation of mintable amounts
- Cumulative precision loss over time
- Reduced token rewards for users

**Proof of Concept:**
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "../src/token/mainnet/INTMAXToken.sol";

contract PrecisionLossTest is Test {
    INTMAXToken public token;
    address public admin = makeAddr("admin");
    
    function setUp() public {
        token = new INTMAXToken(admin, address(0));
    }
    
    function testPrecisionLoss() public {
        // Simulate 1.5 days elapsed (should give partial day rewards)
        uint256 genesisTime = token.GENESIS_TIMESTAMP();
        uint256 elapsedTime = 1.5 days;
        
        vm.warp(genesisTime + uint256(elapsedTime));
        
        // Current implementation (with precision loss)
        uint256 elapsedDaysFloor = elapsedTime / 86400; // = 1 day
        uint256 lostRewards = (elapsedTime % 86400) * 1000e18 / 86400; // Lost partial day
        
        console.log("Elapsed time (seconds):", elapsedTime);
        console.log("Elapsed days (floor):", elapsedDaysFloor);
        console.log("Lost rewards due to precision:", lostRewards);
        
        // Demonstrate the loss
        assertGt(lostRewards, 0, "Should have precision loss");
    }
}
```

**Recommendation:**
Perform multiplication before division:

```solidity
function totalMintableAmount() public view returns (uint256) {
    uint256 elapsedTime = block.timestamp - GENESIS_TIMESTAMP;
    // ...
    totalReward += (elapsedTime * rewardPerDay) / 86400;
}
```

### M-02: Dangerous Strict Equality Check

**Severity:** Medium  
**Location:** `src/minter/Minter.sol#74`

**Description:**
The minter contract uses a strict equality check (`mintedAmount == 0`) which can be bypassed or may fail unexpectedly due to rounding errors or precision issues.

**Impact:**
- Function may fail unexpectedly with very small minting amounts
- Potential for bypassing the zero-mint check
- Reduced reliability of the minting system

**Proof of Concept:**
```solidity
function testStrictEqualityBypass() public {
    // If balance calculation results in tiny amounts due to precision issues,
    // the strict equality check may not catch it
    // Example: balanceAfter = balanceBefore + 1 wei (due to rounding)
    // mintedAmount = 1, which != 0, so check passes despite negligible amount
}
```

**Recommendation:**
Use a threshold-based check instead of strict equality:

```solidity
uint256 constant MINIMUM_MINT_THRESHOLD = 1e12; // 0.000001 tokens

if (mintedAmount < MINIMUM_MINT_THRESHOLD) {
    revert NoTokensToMint();
}
```

## Low/Informational Findings

### I-01: Missing Event Emissions
Several state-changing functions lack event emissions for better transparency and monitoring.

### I-02: Insufficient Input Validation
Some functions could benefit from additional input validation and boundary checks.

## Conclusion

The INTMAX protocol contains several critical security vulnerabilities that require immediate attention. The unchecked token transfer and reentrancy issues pose the highest risk to user funds and protocol integrity. The precision loss and strict equality issues, while less severe, can impact the accuracy and reliability of the reward system.

**Immediate Actions Required:**
1. Fix unchecked token transfers using SafeERC20
2. Implement reentrancy guards on the minter contract
3. Correct the division-before-multiplication precision loss
4. Replace strict equality with threshold-based checks

**Risk Assessment:**
- **High:** 2 findings requiring immediate fixes
- **Medium:** 2 findings affecting system accuracy
- **Overall Risk:** HIGH - Protocol should not go live without addressing H-01 and H-02

The development team should prioritize these fixes and conduct additional testing before any mainnet deployment.
