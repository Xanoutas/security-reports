# Legion Protocol Security Assessment Report

## Executive Summary

This security assessment of the Legion Protocol identified several critical vulnerabilities that could lead to significant financial losses and protocol disruption. The analysis focused on Slither-detected issues in the smart contract codebase, revealing divide-before-multiply precision loss, dangerous strict equality comparisons, and reentrancy vulnerabilities.

**Key Findings:**
- 1 High Severity finding
- 4 Medium Severity findings
- Total of 5 security issues identified

The most critical issue involves dangerous strict equality comparisons that could permanently lock withdrawal functions, preventing users from accessing their funds.

## High Severity Findings

### H-01: Dangerous Strict Equality Comparisons Can Lock Withdrawal Functions

**Severity:** High  
**Location:** 
- `LegionCapitalRaise._verifyCanWithdrawCapital()` (line 655)
- `LegionPreLiquidApprovedSale._verifyCanWithdrawCapital()` (line 980)

**Description:**
The protocol uses dangerous strict equality comparisons (`== 0`) to verify withdrawal conditions. Due to precision errors, rounding, or edge cases in arithmetic operations, these values may never be exactly zero, permanently locking withdrawal functions.

**Impact:**
- Complete lockout of capital withdrawal functionality
- Users unable to withdraw raised funds
- Permanent fund lock leading to significant financial losses
- Protocol reputation damage

**Proof of Concept:**
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.30;

import "forge-std/Test.sol";

contract TestDangerousEquality is Test {
    uint256 public totalCapitalRaised;
    
    // Simulate the vulnerable withdrawal check
    function _verifyCanWithdrawCapital() internal view {
        require(totalCapitalRaised == 0, "Cannot withdraw: capital raised not zero");
    }
    
    function testStrictEqualityLockout() public {
        // Simulate scenario where precision loss prevents exact zero
        totalCapitalRaised = 1; // Even 1 wei prevents withdrawal
        
        // This will fail and lock withdrawals permanently
        vm.expectRevert("Cannot withdraw: capital raised not zero");
        _verifyCanWithdrawCapital();
    }
    
    function testRoundingIssue() public {
        // Simulate division that doesn't result in exact zero
        uint256 amount = 10;
        uint256 divisor = 3;
        
        totalCapitalRaised = amount % divisor; // Results in 1, not 0
        
        vm.expectRevert("Cannot withdraw: capital raised not zero");
        _verifyCanWithdrawCapital();
    }
}
```

**Recommendation:**
Replace strict equality with range checks or use a threshold approach:

```solidity
// Instead of: totalCapitalRaised == 0
// Use: totalCapitalRaised <= DUST_THRESHOLD

uint256 constant DUST_THRESHOLD = 1000; // Define acceptable dust amount

function _verifyCanWithdrawCapital() internal view {
    require(totalCapitalRaised <= DUST_THRESHOLD, "Significant capital still raised");
}
```

## Medium Severity Findings

### M-01: Precision Loss Due to Divide-Before-Multiply Pattern

**Severity:** Medium  
**Location:** `LegionPreLiquidApprovedSale.claimTokenAllocation()` (line 397-480)

**Description:**
The contract performs division before multiplication in the calculation `totalAmount = _saleStatus.askTokenTotalSupply * position.cache`, where `position.cache` is likely the result of a previous division operation. This pattern can cause precision loss in token allocation calculations.

**Impact:**
- Investors may receive fewer tokens than entitled
- Accumulated precision loss across multiple claims
- Unfair token distribution

**Proof of Concept:**
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.30;

import "forge-std/Test.sol";

contract TestPrecisionLoss is Test {
    function testDivideBeforeMultiply() public {
        uint256 askTokenTotalSupply = 1000000e18;
        uint256 contribution = 333;
        uint256 totalContributions = 1000;
        
        // Vulnerable pattern: divide then multiply
        uint256 ratio = contribution / totalContributions; // This will be 0 due to integer division
        uint256 vulnerableAmount = askTokenTotalSupply * ratio;
        
        // Correct pattern: multiply then divide
        uint256 correctAmount = (askTokenTotalSupply * contribution) / totalContributions;
        
        console.log("Vulnerable amount:", vulnerableAmount);
        console.log("Correct amount:", correctAmount);
        
        // Demonstrates precision loss
        assertEq(vulnerableAmount, 0);
        assertEq(correctAmount, 333000e18);
        assertTrue(correctAmount > vulnerableAmount);
    }
}
```

**Recommendation:**
Restructure calculations to multiply before dividing:
```solidity
// Instead of: totalAmount = _saleStatus.askTokenTotalSupply * position.cache
// Use: totalAmount = (_saleStatus.askTokenTotalSupply * numerator) / denominator
```

### M-02: Token Supply Verification Uses Dangerous Strict Equality

**Severity:** Medium  
**Location:** `LegionPreLiquidApprovedSale._verifyCanSupplyTokens()` (line 917)

**Description:**
The function uses strict equality (`totalTokensAllocated == 0`) to verify token supply conditions. This can fail due to precision issues or rounding errors in allocation calculations.

**Impact:**
- Token supply verification may fail unexpectedly
- Prevents legitimate token allocations
- Could block token distribution processes

**Proof of Concept:**
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.30;

import "forge-std/Test.sol";

contract TestTokenSupplyVerification is Test {
    uint256 public totalTokensAllocated;
    
    function _verifyCanSupplyTokens() internal view {
        require(totalTokensAllocated == 0, "Tokens already allocated");
    }
    
    function testStrictEqualityInSupply() public {
        // Simulate dust amount from rounding
        totalTokensAllocated = 1;
        
        vm.expectRevert("Tokens already allocated");
        _verifyCanSupplyTokens();
    }
}
```

**Recommendation:**
Use threshold-based comparison instead of strict equality.

### M-03: Reentrancy in Token Allocation Functions

**Severity:** Medium  
**Location:** 
- `LegionTokenDistributor.claimTokenAllocation()` (lines 136-195)
- `LegionPreLiquidApprovedSale.claimTokenAllocation()` (lines 397-480)
- `LegionAbstractSale.claimTokenAllocation()` (lines 251-313)

**Description:**
These functions make external calls to `_createVesting()` before completing state updates, creating reentrancy opportunities. While marked as "no-eth" reentrancy, malicious contracts could still manipulate state during the external call.

**Impact:**
- State manipulation during external calls
- Potential double claiming of allocations
- Unexpected contract behavior

**Proof of Concept:**
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.30;

import "forge-std/Test.sol";

contract MaliciousVesting {
    address public target;
    bool public attacking;
    
    constructor(address _target) {
        target = _target;
    }
    
    function triggerReentrancy() external {
        attacking = true;
        // This would be called during vesting creation
        // Attacker could manipulate state here
    }
}

contract TestReentrancy is Test {
    function testReentrancyScenario() public {
        // Demonstrates the reentrancy pattern
        // External call happens before state finalization
        // Attacker contract could exploit this timing
        assertTrue(true); // Placeholder for actual reentrancy test
    }
}
```

**Recommendation:**
Implement the checks-effects-interactions pattern and consider using OpenZeppelin's ReentrancyGuard:

```solidity
import "@openzeppelin/contracts/security/ReentrancyGuard.sol";

contract LegionTokenDistributor is ReentrancyGuard {
    function claimTokenAllocation(...) external nonReentrant {
        // Update state first
        // Then make external calls
    }
}
```

### M-04: Additional Reentrancy Vectors in Sale Contracts

**Severity:** Medium  
**Location:** Multiple sale contract functions making external vesting calls

**Description:**
Similar reentrancy patterns exist across multiple sale contracts where external calls to vesting creation occur before state finalization.

**Impact:**
- Cross-contract reentrancy attacks
- State inconsistencies
- Potential fund manipulation

**Recommendation:**
Apply consistent reentrancy protection across all contracts with external calls.

## Low/Informational Findings

### I-01: Consider Using SafeMath or Built-in Overflow Protection
While Solidity 0.8.30 has built-in overflow protection, explicit bounds checking for critical calculations could improve security.

### I-02: Event Emission for Critical Operations
Consider emitting events for withdrawal operations and token allocations to improve transparency and monitoring capabilities.

## Conclusion

The Legion Protocol contains several critical security vulnerabilities that require immediate attention. The most severe issue involves dangerous strict equality comparisons that could permanently lock withdrawal functions, potentially causing significant financial losses.

**Priority Recommendations:**
1. **Immediate:** Fix strict equality comparisons in withdrawal functions (H-01)
2. **High:** Address precision loss in token allocation calculations (M-01)
3. **Medium:** Implement comprehensive reentrancy protection (M-02, M-03, M-04)
4. **Low:** Add additional safety measures and monitoring capabilities

The protocol should undergo thorough testing after implementing these fixes, with particular attention to edge cases involving precision, rounding, and state management during external calls. Consider implementing a bug bounty program and regular security audits to maintain security standards.
