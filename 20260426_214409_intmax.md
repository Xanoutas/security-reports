# Security Audit Report: INTMAX Protocol

## Executive Summary

This security audit of the INTMAX Protocol identified several critical vulnerabilities that pose significant risks to the protocol's security and functionality. The audit focused on the core smart contracts including the token minting system, reward distribution mechanism, and associated governance structures.

**Key Findings:**
- 1 High Severity vulnerability (Unchecked Transfer)
- 3 Medium Severity vulnerabilities (Reentrancy, Dangerous Equality, Precision Loss)

The most critical issue involves unchecked return values in token transfers, which could lead to silent failures and loss of funds. Additionally, reentrancy vulnerabilities and precision loss in calculations present moderate risks that should be addressed promptly.

## High Severity Findings

### H-01: Unchecked Transfer Return Value in BlockBuilderReward

**Severity:** High  
**Location:** `src/block-builder-reward/BlockBuilderReward.sol#145`

**Description:**
The `claimReward()` function in the BlockBuilderReward contract ignores the return value of the `intmaxToken.transfer()` call. Many ERC20 tokens return `false` on failure rather than reverting, which could result in silent failures where users believe they have received rewards when the transfer actually failed.

**Impact:**
- Users may lose rewards due to silent transfer failures
- Protocol state becomes inconsistent with actual token balances
- Potential loss of funds for legitimate reward claimants

**Proof of Concept:**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import "../src/block-builder-reward/BlockBuilderReward.sol";

contract MaliciousToken {
    mapping(address => uint256) public balanceOf;
    
    function transfer(address to, uint256 amount) external returns (bool) {
        // Simulate a token that returns false instead of reverting
        return false;
    }
    
    function mint(address to, uint256 amount) external {
        balanceOf[to] += amount;
    }
}

contract TestUncheckedTransfer is Test {
    BlockBuilderReward reward;
    MaliciousToken token;
    address user = makeAddr("user");
    address admin = makeAddr("admin");
    address rewardManager = makeAddr("rewardManager");
    address contributionContract = makeAddr("contributionContract");
    
    function setUp() public {
        token = new MaliciousToken();
        
        // Deploy and initialize BlockBuilderReward
        reward = new BlockBuilderReward();
        reward.initialize(admin, rewardManager, contributionContract, address(token));
        
        // Set up reward for user
        vm.prank(rewardManager);
        reward.setReward(user, 1000 * 1e18);
        
        // Mint tokens to the reward contract
        token.mint(address(reward), 2000 * 1e18);
    }
    
    function testUncheckedTransferFailure() public {
        uint256 userBalanceBefore = token.balanceOf(user);
        uint256 contractBalanceBefore = token.balanceOf(address(reward));
        
        // User tries to claim reward
        vm.prank(user);
        reward.claimReward(500 * 1e18);
        
        // Verify the transfer failed but the function didn't revert
        assertEq(token.balanceOf(user), userBalanceBefore, "User balance should not change");
        assertEq(token.balanceOf(address(reward)), contractBalanceBefore, "Contract balance should not change");
        
        // However, the reward is marked as claimed in the contract state
        assertEq(reward.claimedAmount(user), 500 * 1e18, "Claimed amount should be updated");
    }
}
```

**Recommendation:**
Use OpenZeppelin's `SafeERC20` library to handle token transfers safely:

```solidity
import "@openzeppelin/contracts/token/ERC20/utils/SafeERC20.sol";

contract BlockBuilderReward {
    using SafeERC20 for IERC20;
    
    function claimReward(uint256 amount) external nonReentrant {
        // ... existing logic ...
        
        // Replace this line:
        // intmaxToken.transfer(_msgSender(), reward);
        
        // With this:
        IERC20(intmaxToken).safeTransfer(_msgSender(), reward);
    }
}
```

## Medium Severity Findings

### M-01: Reentrancy Vulnerability in Minter Contract

**Severity:** Medium  
**Location:** `src/minter/Minter.sol#58-79`

**Description:**
The `mint()` function in the Minter contract is vulnerable to reentrancy attacks. The function reads the balance before making an external call to `intmaxToken.mint()`, then uses this potentially stale balance after the external call completes. An attacker could exploit this by implementing a malicious token contract or hook that calls back into the Minter during the minting process.

**Impact:**
- Potential manipulation of minting calculations
- Inconsistent state during minting operations
- Possible drainage of funds or unauthorized minting

**Proof of Concept:**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import "../src/minter/Minter.sol";

contract ReentrantToken {
    mapping(address => uint256) public balanceOf;
    Minter public minter;
    bool public attacking = false;
    
    function setMinter(address _minter) external {
        minter = Minter(_minter);
    }
    
    function mint(address to) external {
        balanceOf[to] += 1000 * 1e18;
        
        // Reentrant call during minting
        if (!attacking && msg.sender == address(minter)) {
            attacking = true;
            minter.mint(); // Reentrant call
            attacking = false;
        }
    }
}

contract TestReentrancy is Test {
    Minter minter;
    ReentrantToken token;
    address admin = makeAddr("admin");
    address liquidity = makeAddr("liquidity");
    
    function setUp() public {
        token = new ReentrantToken();
        
        minter = new Minter();
        minter.initialize(address(token), liquidity, admin);
        
        token.setMinter(address(minter));
    }
    
    function testReentrancyAttack() public {
        uint256 liquidityBalanceBefore = token.balanceOf(liquidity);
        
        // Trigger the reentrancy attack
        minter.mint();
        
        uint256 liquidityBalanceAfter = token.balanceOf(liquidity);
        
        // The reentrancy could cause unexpected behavior
        console.log("Balance before:", liquidityBalanceBefore);
        console.log("Balance after:", liquidityBalanceAfter);
    }
}
```

**Recommendation:**
Implement the checks-effects-interactions pattern and use OpenZeppelin's `ReentrancyGuard`:

```solidity
import "@openzeppelin/contracts/security/ReentrancyGuard.sol";

contract Minter is ReentrancyGuard {
    function mint() external nonReentrant {
        uint256 balanceBefore = intmaxToken.balanceOf(address(this));
        
        intmaxToken.mint(address(this));
        
        uint256 balanceAfter = intmaxToken.balanceOf(address(this));
        uint256 mintedAmount = balanceAfter - balanceBefore;
        
        // ... rest of the function
    }
}
```

### M-02: Dangerous Strict Equality Check

**Severity:** Medium  
**Location:** `src/minter/Minter.sol#74`

**Description:**
The `mint()` function uses a dangerous strict equality check (`mintedAmount == 0`) which could cause the function to fail unexpectedly. This is particularly problematic when dealing with ERC20 tokens that might have different behaviors or when external factors could influence the minting amount.

**Impact:**
- Function may fail to execute when it should succeed
- Potential DoS if minting consistently returns zero
- Reduced protocol reliability

**Proof of Concept:**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import "../src/minter/Minter.sol";

contract ZeroMintToken {
    mapping(address => uint256) public balanceOf;
    
    function mint(address to) external {
        // Token that sometimes mints zero (due to rate limiting, etc.)
        if (block.timestamp % 2 == 0) {
            balanceOf[to] += 0; // Zero mint
        } else {
            balanceOf[to] += 1000 * 1e18;
        }
    }
}

contract TestDangerousEquality is Test {
    Minter minter;
    ZeroMintToken token;
    address admin = makeAddr("admin");
    address liquidity = makeAddr("liquidity");
    
    function setUp() public {
        token = new ZeroMintToken();
        minter = new Minter();
        minter.initialize(address(token), liquidity, admin);
    }
    
    function testZeroMintReverts() public {
        // Set timestamp to even number to trigger zero mint
        vm.warp(2);
        
        // This should revert due to strict equality check
        vm.expectRevert("mintedAmount == 0");
        minter.mint();
    }
}
```

**Recommendation:**
Replace the strict equality check with a more appropriate condition:

```solidity
function mint() external {
    uint256 balanceBefore = intmaxToken.balanceOf(address(this));
    
    intmaxToken.mint(address(this));
    
    uint256 balanceAfter = intmaxToken.balanceOf(address(this));
    uint256 mintedAmount = balanceAfter - balanceBefore;
    
    // Replace strict equality with a more appropriate check
    if (mintedAmount == 0) {
        return; // Or emit an event instead of reverting
    }
    
    // Continue with normal logic...
}
```

### M-03: Division Before Multiplication Precision Loss

**Severity:** Medium  
**Location:** `src/token/mainnet/INTMAXToken.sol#119-137`

**Description:**
The `totalMintableAmount()` function performs division before multiplication when calculating elapsed days and total rewards. This pattern can lead to precision loss, especially when dealing with timestamps that don't align perfectly with 24-hour boundaries.

**Impact:**
- Inaccurate reward calculations
- Users may receive fewer rewards than intended
- Cumulative precision loss over time

**Proof of Concept:**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";

contract TestPrecisionLoss is Test {
    uint256 constant GENESIS_TIMESTAMP = 1000000;
    uint256 constant SECONDS_PER_DAY = 86400;
    
    function testPrecisionLoss() public {
        // Simulate the problematic calculation
        uint256 currentTime = GENESIS_TIMESTAMP + SECONDS_PER_DAY + 12 * 3600; // 1.5 days
        uint256 rewardPerDay = 1000 * 1e18;
        
        // Current implementation (with precision loss)
        uint256 elapsedDays = (currentTime - GENESIS_TIMESTAMP) / SECONDS_PER_DAY;
        uint256 rewardWithLoss = elapsedDays * rewardPerDay;
        
        // Correct implementation (without precision loss)
        uint256 rewardCorrect = ((currentTime - GENESIS_TIMESTAMP) * rewardPerDay) / SECONDS_PER_DAY;
        
        console.log("Elapsed seconds:", currentTime - GENESIS_TIMESTAMP);
        console.log("Elapsed days (truncated):", elapsedDays);
        console.log("Reward with precision loss:", rewardWithLoss);
        console.log("Correct reward:", rewardCorrect);
        console.log("Lost reward:", rewardCorrect - rewardWithLoss);
        
        // The difference shows the precision loss
        assertLt(rewardWithLoss, rewardCorrect, "Precision loss occurred");
    }
}
```

**Recommendation:**
Reorder the calculation to perform multiplication before division:

```solidity
function totalMintableAmount() public view returns (uint256) {
    if (block.timestamp <= GENESIS_TIMESTAMP) {
        return 0;
    }
    
    uint256 totalReward = 0;
    uint256 currentTime = block.timestamp;
    
    for (uint256 i = 0; i < rewardSchedule.length; i++) {
        RewardPeriod memory period = rewardSchedule[i];
        
        if (currentTime <= period.startTime) {
            break;
        }
        
        uint256 endTime = (i == rewardSchedule.length - 1) ? currentTime : 
                         min(period.endTime, currentTime);
        
        if (endTime > period.startTime) {
            // Perform multiplication before division to avoid precision loss
            uint256 elapsedSeconds = endTime - period.startTime;
            totalReward += (elapsedSeconds * period.rewardPerDay) / 86400;
        }
    }
    
    return totalReward;
}
```

## Low/Informational Findings

### L-01: Missing Event Emissions

**Severity:** Low  
**Location:** Multiple locations

**Description:**
Several critical functions lack proper event emissions, making it difficult to track important state changes off-chain.

**Recommendation:**
Add appropriate events for all state-changing functions, particularly in reward claiming and minting operations.

### L-02: Lack of Input Validation

**Severity:** Low  
**Location:** Multiple locations

**Description:**
Several functions lack proper input validation, which could lead to unexpected behavior or gas waste.

**Recommendation:**
Add comprehensive input validation for all external functions, including zero address checks and reasonable bounds checking.

## Conclusion

The INTMAX Protocol contains several security vulnerabilities that require immediate attention. The unchecked transfer return value represents the highest risk and should be addressed as a priority. The reentrancy vulnerability and precision loss issues also pose significant risks to protocol integrity and user funds.

**Immediate Actions Required:**
1. Implement `SafeERC20` for all token transfers
2. Add reentrancy protection to the Minter contract
3. Fix precision loss in reward calculations
4. Replace dangerous equality checks with more robust conditions

**Recommendations:**
- Conduct thorough testing of all fixes
- Implement comprehensive monitoring and alerting
- Consider formal verification for critical mathematical operations
- Regular security audits as the protocol evolves

The protocol shows promise but requires these security improvements before mainnet deployment or handling significant value.
